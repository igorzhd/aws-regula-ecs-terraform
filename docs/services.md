# Services

---

## Overview

The platform runs two application services, both containerized and deployed as a single ECS task:

| Service | Language / Framework | Port | Role |
|---|---|---|---|
| `python-api` | Python / FastAPI | 8001 | Orchestrates the verification pipeline; the only service exposed to the internet |
| `regula-docreader` | Regula SDK (closed-source, containerized) | 8080 | ML-based document recognition engine; accessible only via localhost within the task |

The frontend is a static site (HTML/CSS/JS) deployed to S3 and served via CloudFront. It calls the Python API over HTTPS.

---

## Python API

### What it does

The Python API is the orchestration layer. It receives uploaded document images from the frontend, routes them through the Regula engine, parses the results, stores session data, and returns structured responses.

It is the only service that talks to everything else: Regula (via localhost), RDS (session storage), and S3 (image storage). The frontend never calls Regula directly.

### Request flow

```
POST /process  (image upload from frontend)
  │
  ├── regula.py    — encode image as base64, POST to Regula at localhost:8080
  │                  receive large JSON (~50–200 KB per document)
  │
  ├── parser.py    — extract from Regula's response:
  │                  • document type (result_type 9)
  │                  • image quality checks (result_type 30): focus, glare, resolution, angle
  │                  • text fields (result_type 36): name, DOB, document number, expiry
  │                    each field includes all sources (MRZ, VISUAL, BARCODE) and cross-validation
  │                  • document crop images (result_type 37, fieldType 207)
  │
  ├── storage.py   — save crop images + raw Regula JSON to S3 (or local disk in dev)
  │
  ├── models.py    — save session record to PostgreSQL (transaction_id, timestamps, field results)
  │
  └── return ProcessResponse (structured JSON to frontend)
```

### API Endpoints

| Method | Path | Description |
|---|---|---|
| `POST` | `/process` | Submit document image(s); returns full verification result |
| `GET` | `/sessions` | List all processed sessions |
| `GET` | `/sessions/{id}` | Retrieve full details of one session |
| `GET` | `/sessions/{id}/download` | Stream raw Regula JSON as file download |
| `GET` | `/sessions/{id}/image/{page}` | Return presigned S3 URL for document crop image |
| `GET` | `/health` | Health check (returns `{"status": "ok"}`) — used by ALB |

Interactive API documentation is available at `/docs` (Swagger UI, auto-generated by FastAPI) in both local and deployed environments.

### Key source files

| File | Responsibility |
|---|---|
| `main.py` | FastAPI application, route definitions, request lifecycle |
| `regula.py` | HTTP client for Regula SDK; builds request payload, handles response |
| `parser.py` | Extracts structured data from Regula's raw JSON response |
| `storage.py` | Dual-mode file storage: local disk (dev) or S3 (production) |
| `models.py` | SQLAlchemy ORM models — defines `sessions` table structure |
| `schemas.py` | Pydantic schemas — validates API request/response shapes |
| `config.py` | Reads environment variables; single source of truth for all config |

### Configuration

The API is configured entirely through environment variables, making the same Docker image run identically in local development and on AWS — only the variable values change.

| Variable | Local development | Production (AWS) |
|---|---|---|
| `REGULA_URL` | `http://regula:8080` (Docker Compose service name) | `http://localhost:8080` (same ECS task) |
| `DATABASE_URL` | `postgresql://postgres:5432/docverify` | RDS endpoint injected by ECS |
| `DB_USERNAME` | `postgres` (Docker Compose) | From SSM via ECS `secrets` block |
| `DB_PASSWORD` | `postgres` (Docker Compose) | From SSM via ECS `secrets` block |
| `S3_BUCKET_NAME` | Not used | Images S3 bucket name |
| `STORAGE_MODE` | `local` | `s3` |

### Technology stack

| Package | Purpose |
|---|---|
| FastAPI | Web framework — async-native, auto-generates OpenAPI docs |
| SQLAlchemy | ORM — Python classes mapped to PostgreSQL tables |
| asyncpg | Async PostgreSQL driver — required for FastAPI's non-blocking architecture |
| Alembic | Database migrations — tracks schema changes over time |
| Pydantic | Request/response validation — schemas defined in `schemas.py` |

| uvicorn | ASGI server — runs the FastAPI application |

## Regula Document Reader

### What it is

Regula Forensics Document Reader is a commercial document recognition SDK. It uses template-based recognition to process images of identity documents and returns structured results including document type identification, machine-readable zone (MRZ) parsing, visual field recognition, barcode reading, and image quality analysis. Regula maintains a database of document templates covering hundreds of document types from different countries, which is what drives recognition.

It ships as a Docker container from Docker Hub (`regulaforensics/docreader`) and exposes an HTTP API on port 8080.

### Integration approach

Regula runs as a sidecar container in the same ECS task as the Python API. The two containers share a network namespace, so the Python API communicates with Regula via `http://localhost:8080` — no service discovery, no security group rules, no load balancer involved.

This keeps Regula completely internal: it is not registered with any target group, not accessible via the ALB, and not reachable from outside the task.

### Image and resource requirements

The Regula container image is approximately 1 GB compressed. However, the runtime memory requirement is significantly higher — the process loads document templates and supporting data into memory at startup.

This was discovered through iterative testing (see [challenges-and-decisions.md](challenges-and-decisions.md#21-regula-container-oom--exit-code-137)). The image must be mirrored to a private ECR repository before deployment — pulling from Docker Hub at task startup creates a dependency on Docker Hub availability and routes traffic through the NAT Gateway on every task launch. An OOM crash loop while pulling from Docker Hub was the root cause of the NAT Gateway cost incident (see [postmortem-nat-gateway.md](postmortem-nat-gateway.md)).

**An additional operational risk:** Regula may update the `latest` tag on Docker Hub with a new version that has changed memory requirements or API behavior. Pinning to a specific image digest in ECR eliminates this risk — you control exactly which version is running.

### Resource requirements

| Resource | Configured | Notes |
|---|---|---|
| Memory | 8192 MB (8 GB) | Shared across both containers in the task |
| CPU | 2048 units (2 vCPU) | Shared across both containers in the task |
| Startup time | ~60–120 seconds | Templates load before first request |

### Regula API request format

The Python API sends a multipart JSON request to Regula:

```json
{
  "processParam": {
    "scenario": "FullProcess",
    "resultTypeOutput": [9, 30, 36, 37]
  },
  "List": [
    {
      "ImageData": {
        "image": "<base64-encoded image data>"
      },
      "light": 6,
      "page_idx": 0
    }
  ]
}
```

`resultTypeOutput` specifies which result types to return:
- `9` — OneCandidate (document type identification)
- `30` — ImageQualityCheckList (focus, glare, resolution, perspective)
- `36` — Text (field values from MRZ, visual zone, barcode, with cross-source comparison)
- `37` — Images (cropped document images)

## Frontend

### What it is

A static web application — plain HTML, CSS, and JavaScript with no build step. Deployed to an S3 bucket and served globally via CloudFront.

### Files

| File | Purpose |
|---|---|
| `index.html` | Main page structure |
| `styles.css` | Styling |
| `app.js` | Document upload, API calls, result rendering |
| `config.js` | API base URL configuration — the only environment-specific file |

### `config.js` — the critical environment file

```javascript
// config.js
const CONFIG = {
  API_BASE_URL: "https://api.example.com"   // production
  // API_BASE_URL: "http://localhost:8001"  // local development
};
```

This file is the only difference between a local development build and a production deployment. In production, it must point to the CloudFront/ALB domain, not localhost. It is synced to S3 as part of the CI/CD pipeline.

**Important:** After updating `config.js` in S3, a CloudFront cache invalidation must be triggered. CloudFront caches responses from S3 at its edge nodes; without invalidation, stale versions of `config.js` can be served to users for hours. The GitHub Actions pipeline handles this automatically.

### Deployment

The frontend is deployed by the CI/CD pipeline (`deploy.yml`):

```bash
# Sync all frontend files to S3
aws s3 sync services/frontend/ s3://<bucket-name>/ --delete

# Invalidate CloudFront cache
aws cloudfront create-invalidation \
  --distribution-id <distribution-id> \
  --paths "/*"
```

The `--delete` flag removes files from S3 that no longer exist locally. The `"/*"` invalidation path clears all cached files from all edge locations.

### CloudFront configuration

The CloudFront distribution serves both the frontend and the API from a single domain:

| Path pattern | Origin | Notes |
|---|---|---|
| `/api/*` | ALB | API routes with `/api/` prefix |
| `/health` | ALB | Direct health check path |
| `/process*` | ALB | Document processing endpoint |
| `/sessions*` | ALB | Session history endpoints |
| `*` (default) | S3 | All frontend static files |

The S3 origin uses Origin Access Control (OAC) — requests are signed with SigV4, and the S3 bucket policy allows `s3:GetObject` only from this specific CloudFront distribution. Direct S3 URLs return 403.

## Local Development

Run the full stack locally with Docker Compose:

```bash
cd services/python-api
docker compose up
```

Docker Compose starts three containers:
- `regula` — Regula DocReader at `http://localhost:8080`
- `postgres` — PostgreSQL at `localhost:5432`
- `python-api` — FastAPI at `http://localhost:8001`

Access Swagger UI at `http://localhost:8001/docs` to test all API endpoints directly without a frontend.

The `STORAGE_MODE=local` setting stores document images to `services/python-api/local_storage/` instead of S3.
